# Cotext와 Neural Link, 그리고 세컨드 브레인 방향성

## 목적

이 문서는 아래 내용을 팀 AI 개발자가 빠르게 이해할 수 있도록 정리한 한국어 설명서입니다.

- Cotext의 기본 구조
- Neural Link의 구조와 작동 원리
- Obsidian과의 차이
- 앞으로 knowledge graph와 자동 지식망 확장을 통해
  - 세컨드 브레인
  - 인공 지식망
  - 에이전트 기반 외부 사고 시스템
  으로 발전할 수 있는 방향성

## 1. Cotext는 무엇인가

Cotext는 단순한 채팅 앱이나 메모 앱이 아닙니다.

핵심은 다음입니다.

- GitHub 저장소를 하나의 협업형 컨텍스트 공간으로 사용한다
- 사람과 에이전트가 같은 맥락 위에서 읽고 쓴다
- 대화, 판단, 문맥, 지식 연결을 모두 repo 안에 남긴다

즉, Cotext는 "repo 기반 업무 기억 시스템"에 가깝습니다.

## 2. Cotext의 기본 구조

Cotext는 대략 아래 계층으로 이해하면 됩니다.

### 2.1 Repository Layer

- 최종 source of truth는 GitHub repo입니다
- 채팅 내용, 컨텍스트 파일, 그래프 파일, 가이드 문서가 repo 안에 존재합니다

### 2.2 Workspace Layer

- 하나의 workspace는 하나의 GitHub repo와 연결됩니다
- 팀원은 workspace 초대를 통해 같은 repo 컨텍스트에 참여합니다
- 초대된 멤버도 owner와 같은 내용을 읽고 수정할 수 있어야 합니다

### 2.3 Room Layer

- room은 특정 경로, 기능, 주제에 대응되는 대화/기록 스트림입니다
- 각 room은 시간순 markdown block들의 집합입니다

### 2.4 Draft / Push Layer

- 로컬 수정은 draft로 남을 수 있습니다
- push를 해야 Git 기반 공유 상태로 올라갑니다
- 즉, "로컬 사고 중인 상태"와 "공유된 팀 상태"가 분리됩니다

### 2.5 Agent Layer

- 에이전트도 같은 컨텍스트를 읽습니다
- 필요하면 블록 단위로 다시 써서 room에 저장할 수 있습니다
- 이때 provenance와 author가 남습니다

## 3. Cotext의 최소 단위: 블록

Cotext에서 가장 중요한 단위는 "파일"이 아니라 "블록"입니다.

블록 예시는 아래와 같습니다.

```md
## 2026-06-16 18:30
<!-- source: chatgpt; author: alvintracer -->

여기에 실제 내용이 들어간다.
```

이 블록에는 3가지 핵심 정보가 있습니다.

- `timestamp`
  - 언제 생성된 기록인지
- `source`
  - 누가 생성했는지
  - 예: `me`, `chatgpt`, `claude`
- `author`
  - 실제 GitHub 작성자 계정이 누구인지

이 구조가 중요한 이유는, Cotext가 단순 텍스트 저장이 아니라
"기록 + 출처 + 작성자 + 후속 지식 연결"까지 고려한 시스템이기 때문입니다.

## 4. Neural Link는 무엇인가

Neural Link는 Cotext 위에 올라가는 의미 네트워크 계층입니다.

Cotext가 시간순 기록을 담당한다면,
Neural Link는 그 기록을 의미 단위로 연결합니다.

쉽게 말하면:

- 채팅 블록은 시간축 기억
- Neural Link는 의미축 기억

입니다.

## 5. Neural Link의 구조

Neural Link는 아래 요소들로 구성됩니다.

### 5.1 Node

- 하나의 중요한 블록을 "의미 단위"로 승격한 것
- 블록에 라벨을 붙여 생각 단위로 다룹니다

예:

- 가격 정책 v2
- IA 구조 개편
- 초대 워크스페이스 권한 문제

### 5.2 Cluster

- 관련 node들을 묶는 그룹
- 단순 태그보다 더 구조화된 개념 묶음입니다

예:

- pricing
- onboarding
- workspace-sharing
- neural-link

### 5.3 Edge

- node와 node 사이의 명시적 관계

예:

- relates
- supports
- supersedes

즉, "비슷하다" 수준이 아니라
"이게 이걸 보강한다", "이게 이걸 대체했다" 같은 구조를 표현합니다.

### 5.4 Graph Files / Index

- `.cotext/neural.json`
  - 그래프의 실제 구조 데이터
- `NEURAL_INDEX.md`
  - 에이전트가 쉽게 읽을 수 있도록 정리된 그래프 요약
- Supabase 파생 인덱스
  - 검색, 관련 노드 찾기, 클러스터 탐색 등에 사용

## 6. Neural Link의 작동 원리

전체 흐름은 아래처럼 이해하면 됩니다.

1. 사람이나 에이전트가 room에 블록을 쌓는다
2. 중요한 블록을 node로 승격한다
3. node를 cluster에 넣는다
4. node끼리 edge로 연결한다
5. graph를 검색/요약/grounding에 사용한다

즉,

`대화 -> 블록 -> 노드 -> 클러스터/엣지 -> 그래프 -> 검색/추론/재사용`

의 흐름입니다.

## 7. Cotext와 Obsidian의 차이

겉으로 보면 둘 다 markdown과 graph를 다루는 것처럼 보일 수 있습니다.
하지만 철학과 구조가 다릅니다.

### 7.1 중심 단위가 다르다

- Obsidian: note/file 중심
- Cotext: repo -> workspace -> room -> block 중심

Obsidian은 문서를 쓰는 시스템에 가깝고,
Cotext는 "업무 맥락을 기록하고 공유하는 시스템"에 가깝습니다.

### 7.2 협업 모델이 다르다

- Obsidian: 기본적으로 개인 지식 관리에 강함
- Cotext: 팀 + 에이전트 협업을 전제로 설계

### 7.3 write-back 구조가 다르다

- Obsidian: 사람이 수동 편집하는 흐름이 기본
- Cotext: 에이전트도 같은 room에 결과를 다시 쓸 수 있음

### 7.4 provenance와 author가 다르다

- Obsidian: 작성자/출처가 파일 수준에서 암묵적일 때가 많음
- Cotext: 블록 단위로 `source`와 `author`가 명시됨

### 7.5 graph의 의미가 다르다

- Obsidian graph: note 간 링크 그래프에 가까움
- Neural Link: 실제 대화 블록에 앵커된 의미 그래프

즉, Cotext의 graph는 "문서 링크 시각화"보다
"생각 단위의 관계망"에 더 가깝습니다.

## 8. 왜 이 구조가 중요한가

일반 채팅 시스템은 대화는 남기지만 구조를 잃습니다.
일반 노트 시스템은 문서는 남기지만 실행 맥락을 잃습니다.
일반 코드 저장소는 결과물은 남기지만 사고 과정을 잃습니다.

Cotext는 이 셋을 합치려는 시도입니다.

- 대화 기록
- 판단 과정
- 문맥
- 출처
- 작성자
- 의미 연결
- Git 기반 내구성
- 에이전트 참여

이것이 모이면 단순 기록이 아니라 "운영 기억"이 됩니다.

## 9. 세컨드 브레인으로 가는 방향

Cotext가 세컨드 브레인이 되려면 핵심은 "많이 저장"이 아닙니다.
핵심은 "일을 하면서 지식 구조가 자연스럽게 축적"되는 것입니다.

필요한 축은 3개입니다.

### 9.1 Memory Capture

- 중요한 업무 맥락이 블록으로 남아야 함
- 결정, 실패, 대안, 판단 이유가 기록되어야 함

### 9.2 Semantic Organization

- 중요한 블록이 node가 되어야 함
- 관련 node들이 cluster/edge로 연결되어야 함

### 9.3 Agent Participation

- 에이전트가 단순 답변만 하는 것이 아니라
  - 정리하고
  - 연결하고
  - 중복을 찾고
  - 누락된 관계를 제안하고
  - 요약과 재구성을 수행해야 함

이렇게 되면 Cotext는 단순한 context pool이 아니라
프로젝트의 외부 기억 장치가 됩니다.

## 10. 인공 지식망으로 가는 방향

Neural Link가 진짜 강해지는 시점은
그래프가 수동 편집 자료를 넘어서 "살아 있는 지식망"이 될 때입니다.

예를 들면 앞으로 에이전트가 아래를 할 수 있습니다.

- 새 블록에서 node 후보 자동 탐지
- cluster 추천
- edge 추천
- supersedes 관계 자동 탐지
- 중복 개념 병합 제안
- 서로 다른 room 사이의 숨은 연결 발견
- 오래된 지식과 최신 지식의 충돌 탐지

이 단계가 되면 그래프는 단순히 사람이 그려 넣는 맵이 아니라
사람과 에이전트가 함께 증식시키는 인공 지식망이 됩니다.

## 10.5 왜 외부 지식망이 중요한가

지금 LLM 시스템의 가장 큰 한계 중 하나는,
유용한 컨텍스트가 대부분 모델 내부 세션 안에 블랙박스처럼 갇혀 있다는 점입니다.

이 상태에서는 아래 문제가 생깁니다.

- 사람이 모델이 무엇을 근거로 답했는지 쉽게 확인하기 어렵다
- 팀이 그 컨텍스트를 함께 재사용하거나 수정하기 어렵다
- 세션이 바뀌면 맥락이 쉽게 끊긴다
- 중요한 연결관계가 암묵적으로만 남고 공동 지식이 되지 못한다

세컨드 브레인과 외부 지식망의 의미는 여기서 나옵니다.

Cotext와 Neural Link는 컨텍스트를 특정 LLM 내부에 가두지 않고
외부의 읽을 수 있는 구조로 꺼내 놓습니다.

그 결과:

- 사람도 쉽게 이해할 수 있고
- 사람도 쉽게 편집할 수 있고
- 사람도 쉽게 연결할 수 있고
- 팀의 공동 지식망으로 축적할 수 있습니다

동시에:

- 에이전트도 쉽게 이해할 수 있고
- 에이전트도 쉽게 편집할 수 있고
- 에이전트도 쉽게 연결할 수 있고
- 에이전트가 그 지식망을 계속 증강할 수 있습니다

즉, 핵심 가치는 단순히 "좋은 메모 시스템"이 아닙니다.

핵심은:

- 개별 LLM 안에 갇혀 있던 컨텍스트를 외부화하고
- 사람과 에이전트가 함께 다룰 수 있는 구조로 만들고
- 그것을 팀 단위 공동 지식망으로 축적하는 것

입니다.

이 관점에서 Cotext와 Neural Link는
"블랙박스 컨텍스트를 개방형 지식망으로 바꾸는 시스템"이라고 볼 수 있습니다.

## 11. Knowledge Graph 연계 방향

향후에는 Neural Link를 일반적인 knowledge graph 체계와 더 강하게 연결할 수 있습니다.

예상 방향:

- node를 더 안정적인 개체 ID로 관리
- edge 타입을 더 정교하게 세분화
- cluster를 ontology-lite처럼 다루기
- 외부 문서/코드/이슈/PR과 연결
- agent가 graph를 탐색하며 reasoning chain을 구성

그러면 Cotext의 graph는 단순 시각화 자료가 아니라,
실제 추론과 검색과 제안의 기반이 됩니다.

## 12. 장기적으로 가능한 모습

장기적으로 Cotext는 아래 방향으로 갈 수 있습니다.

- 팀의 작업 기억 시스템
- 프로젝트의 장기 맥락 저장소
- 사람과 에이전트의 공동 사고 공간
- 지속적으로 확장되는 의미 그래프
- 외부화된 팀 세컨드 브레인
- 반자동 인공 지식망

핵심은 "문서 앱"이 되는 것이 아닙니다.
핵심은 "일하는 과정이 지식망으로 전환되는 시스템"이 되는 것입니다.

## 13. 앞으로 유지해야 할 원칙

이 방향을 유지하려면 아래 원칙이 중요합니다.

- 블록 단위 identity가 안정적이어야 한다
- provenance와 author는 항상 명시되어야 한다
- graph 구조는 repo 안에서 추적 가능해야 한다
- agent의 변경은 검토 가능해야 한다
- Git 기반 내구성은 계속 핵심이어야 한다
- 자동화는 사람의 판단을 보조해야지 숨겨서 대체하면 안 된다

이 원칙을 지키면 Cotext는 신뢰를 잃지 않으면서도
점점 더 강한 외부 사고 시스템으로 발전할 수 있습니다.

## 한 줄 요약

Cotext는 "repo 기반 협업 기억 시스템"이고,
Neural Link는 그 위에 쌓이는 "의미 그래프 계층"이며,
장기적으로는 사람과 에이전트가 함께 키우는 세컨드 브레인 / 인공 지식망으로 발전할 수 있습니다.
